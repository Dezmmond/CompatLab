# CompatLab ArtifactDoctor 0.5

Release date: 2026-06-28

## Summary

CompatLab ArtifactDoctor 0.5 adds Docker image target profile generation.

This release can generate a CompatLab YAML target profile from a Docker image
without requiring Python, `readelf`, `ldconfig`, or development tooling inside
the image. CompatLab uses the host Docker CLI to export a temporary container
rootfs, parses that tar archive from Python, extracts selected libraries, and
runs host-side `readelf` for symbol-version detection.

## Added

- Rootfs tar parsing helpers for exported Docker filesystems:
  - read file content by path;
  - parse `/etc/os-release`;
  - list library basenames from common library directories;
  - detect common dynamic linker paths;
  - extract candidate libraries to temporary host files.
- Docker CLI wrapper over the existing safe command runner:
  - `docker image inspect`;
  - optional `docker pull`;
  - `docker create`;
  - `docker export`;
  - cleanup through `docker rm -f`.
- Docker image detection flow:
  - Docker image metadata;
  - exported rootfs facts;
  - dynamic linkers;
  - provided libraries;
  - `GLIBC_*`, `GLIBCXX_*`, and `CXXABI_*` versions from extracted libraries.
- `compatlab profiles generate --from-image IMAGE --name NAME --output PATH`.
- Optional Docker generation flags:
  - `--pull`;
  - `--platform PLATFORM`.
- `compatlab profiles detect --from-image IMAGE --json PATH` for raw image
  facts export.
- Docker-source metadata in generated profiles:
  - source image;
  - source image ID;
  - platform;
  - Docker rootfs export detection backend.
- Unit tests for rootfs tar parsing, Docker command orchestration, image facts,
  CLI behavior, and generated Docker profile metadata.

## Changed

- `compatlab profiles generate` now requires exactly one source selector:
  - `--from-current`;
  - `--from-image IMAGE`.
- Docker architecture metadata such as `amd64` is normalized to the existing
  profile architecture naming style, for example `x86_64`.
- The existing `SystemFacts -> TargetProfile` generation path is reused for
  Docker images instead of introducing a second profile schema.

## Current Behavior

Users can generate a profile from a locally available Docker image:

```bash
uv run compatlab profiles generate \
  --from-image ubuntu:22.04 \
  --name ubuntu-2204-docker \
  --output /tmp/ubuntu-2204-docker.yaml
```

The generated profile validates with the existing validator:

```bash
uv run compatlab profiles validate /tmp/ubuntu-2204-docker.yaml
```

It can then be used by compatibility comparison:

```bash
uv run compatlab compare ./dist/my-app --target-file /tmp/ubuntu-2204-docker.yaml
```

Raw Docker-image detection facts can be exported for debugging:

```bash
uv run compatlab profiles detect --from-image ubuntu:22.04 --json /tmp/image-facts.json
```

## Not Included Yet

- Runtime profile presets with package installation.
- Installing packages inside Docker images.
- Mutating Docker images.
- Generating custom Dockerfiles.
- Rootfs scanning as a public standalone feature.
- Scanning artifacts inside Docker containers.
- Recursive dependency resolution.
- Local filesystem dependency resolution for scanned artifacts.
- Wheel, RPM, or DEB package scanning.
- HTML reports.
- Web interface, database, daemon, or server mode.
- Automatic patching through `patchelf`.
- Go helper implementation.
